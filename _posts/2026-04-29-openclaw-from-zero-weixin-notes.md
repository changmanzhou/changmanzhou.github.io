这篇笔记记录我在阿里云 Ubuntu 服务器上部署 OpenClaw，并接入 `openclaw-weixin` 插件时踩过的一些真实坑。目标不是复述官方文档，而是把一条从 0 到可用、能稳定跑起来、能远程访问控制台、能接微信的链路讲清楚。

很多问题并不是“命令不会敲”，而是表面看起来已经成功，实际上后台进程、浏览器安全策略、SSH 隧道、插件状态这些环节任何一个没处理好，都会让你感觉它“明明启动了，但就是不能用”。

# 一、OpenClaw 基础介绍

OpenClaw 可以理解成一个本地或远程运行的 AI Agent 网关系统。它负责管理模型配置、控制台、插件、通道，以及具体的 Agent 执行。

它的几个核心概念可以先这样理解：

- `Gateway`：核心服务进程。控制台、登录状态、插件、channel 等能力基本都挂在它上面。
- `Agent`：真正执行任务的智能体，依赖你配置的模型。
- `Channel`：外部接入通道，比如微信。消息通过 channel 进入，再由 agent 处理。

默认情况下，OpenClaw 控制台监听端口是：

```text
18789
```

这个端口后面会频繁用到。无论是本地访问、SSH 隧道、Nginx 反代，核心都绕不开它。

# 二、服务器部署流程

下面以阿里云 Ubuntu 服务器为例，整理一条更稳的部署方式。

## 1. 安装 OpenClaw

先确认机器上已经有 Node.js，并且版本足够新。如果你是刚开的 Ubuntu 服务器，建议先把 Node.js 环境准备好，再安装 OpenClaw。

安装完成后，可以先确认命令存在：

```bash
openclaw --help
```

如果这里就提示 `command not found`，先不要继续往后排查微信或控制台，说明基础安装还没通。

## 2. 启动 gateway

最直接的启动方式是：

```bash
openclaw gateway
```

这条命令的作用是前台启动 Gateway。第一次调试时很好用，因为日志会直接打印在终端里，出错也容易看见。

但它有一个特别容易踩的坑：如果你是在 VSCode Remote-SSH 的终端、普通 SSH 会话，或者某个会自动断开的 shell 里运行它，那么一旦终端关闭、VSCode 断开、SSH 退出，Gateway 往往也就一起停了。

这就是很多人会遇到的情况：

- 浏览器里刚开始能打开控制台
- 过一会儿突然打不开
- 微信消息也不再响应
- 误以为是 token、端口、插件又坏了

其实根因只是：`openclaw gateway` 跟着会话一起结束了。

## 3. 后台运行（systemd）

真正适合长期部署的方式，是用 `systemd --user` 让 Gateway 常驻。

```bash
systemctl --user start openclaw-gateway
systemctl --user enable openclaw-gateway
loginctl enable-linger root
```

这里要理解一下为什么要这么做。

`systemctl --user start openclaw-gateway`

作用是启动 OpenClaw 的用户级服务，让 Gateway 从“当前 shell 进程”变成“systemd 管理的后台服务”。

`systemctl --user enable openclaw-gateway`

作用是开机或用户会话恢复时自动拉起，不用每次手动敲命令。

`loginctl enable-linger root`

这一条很关键。它的作用是允许该用户在没有登录会话时，用户级 systemd 也继续运行。否则你虽然用了 `systemctl --user`，但 SSH 退出后服务仍然可能被系统回收。

如果你部署时用的不是 `root`，这里把用户名换成你自己的用户即可。

## 为什么不能只用 `openclaw gateway`

因为这条命令默认是前台运行，生命周期依赖当前终端。调试时方便，生产上不稳。

一个很典型的真实坑是：你在 VSCode 远程终端里把 Gateway 跑起来了，当时看控制台一切正常；然后你把电脑盖上、网络切换、VSCode 断开，服务器上的 OpenClaw 也就没了。第二天回来你会以为“网页打不开”“微信失效”“token 失效”，实际上只是服务没在跑。

所以建议分两阶段：

1. 首次调试时，用 `openclaw gateway` 看日志。
2. 确认配置没问题后，切换到 `systemd` 常驻。

# 三、模型配置

OpenClaw 的可用性很大程度取决于模型配置是否正确。很多“Agent 没反应”或者“响应很慢”的问题，最后都能追溯到模型没配好、模型能力不匹配，或者主模型挂了但没有 fallback。

## 如何配置模型

常见做法是配置你已经能稳定调用的模型，比如：

- `qwen`
- `glm`

不同环境里的 provider 名称和 model id 可能不完全一样，但核心思路都是先把默认 agent 使用的主模型指向一个可用模型。

## 模型切换方法

可以用下面这条命令切换默认主模型：

```bash
openclaw config set agents.defaults.model.primary xxx
```

这里的 `xxx` 替换成你实际想用的模型名。

如果你已经配好了多个模型，切换时优先确认两件事：

- 这个模型本身在当前 provider 下确实存在
- 对应的 API Key、Base URL、限额状态都正常

否则你会看到“看起来配置成功了”，但实际 Agent 仍无法执行。

## 推荐模型策略

比较实用的一种策略是：

- 主模型用 `flash` / 更快的模型：降低延迟，适合微信这种即时对话场景
- fallback 用更稳或更强的模型：在主模型失败、超时或限速时兜底

如果你的重点是响应速度，微信接入后优先保证“能及时回话”。如果你的重点是复杂任务质量，再把高质量模型作为 fallback 或部分 Agent 的专用模型。

## fallback 机制说明

fallback 可以理解为“主模型失败时的备用模型”。它不是为了炫技，而是为了提高系统的韧性。

真实使用里常见的问题包括：

- 主模型偶发超时
- provider 返回 429
- 某个模型临时不可用
- 模型名改了但配置还没同步

如果没有 fallback，这些问题会直接表现为 Agent 卡死、失败或无响应；如果有 fallback，至少系统还能继续工作。

# 四、微信接入

这一部分通常是部署过程中最容易让人“明明步骤都对了，怎么还是不通”的环节。

## 1. 插件安装

先安装微信插件：

```bash
openclaw plugins install "@tencent-weixin/openclaw-weixin@latest"
```

如果安装过程中出现网络慢、拉包失败、偶发超时，先检查服务器本身的外网连通性。很多时候不是 OpenClaw 本身有问题，而是云服务器访问 npm 相关资源不稳定。

## 2. 登录绑定

插件装好后，执行：

```bash
openclaw channels login --channel openclaw-weixin
```

这个命令会触发微信 channel 的登录绑定流程。

## 3. 扫码流程说明

通常你会在终端或控制台里拿到登录二维码，然后用微信扫码完成绑定。

这里有两个容易混淆的点：

第一，扫码成功不等于 Gateway 已经稳定运行。

如果你是用前台 `openclaw gateway` 启动的，扫码后短时间内也许一切正常，但只要终端断掉，channel 状态也会失效。

第二，扫码成功不等于浏览器控制台一定能从公网直接打开。

微信登录是 channel 能力，控制台访问又是另一层浏览器安全策略问题，这两个不要混在一起排查。

## 4. 成功标志

登录完成后，可以用下面的命令检查 channel 状态：

```bash
openclaw channels list
```

如果这里能看到 `openclaw-weixin` 已注册、状态正常，说明微信通道至少已经完成绑定。

# 五、远程访问控制台

这一部分是整个部署里最容易反复踩坑的地方，也是最需要解释“为什么”的部分。

## 1. 本地访问（localhost）

如果你直接在服务器本机打开浏览器，访问：

```text
http://localhost:18789
```

通常是可以工作的。

本地 `localhost` 之所以特殊，不是因为 token 更高级，也不是因为 OpenClaw 对它做了白名单，而是因为浏览器把 `localhost` 视为更安全的上下文之一。

## 2. SSH 隧道方式（推荐）

如果你在自己电脑上访问远程服务器，最简单稳妥的方式是 SSH 隧道：

```bash
ssh -L 18789:127.0.0.1:18789 root@服务器IP
```

建立隧道后，在你自己的电脑浏览器里访问：

```text
http://localhost:18789
```

这时候虽然实际服务跑在远程服务器上，但浏览器看到的是本地 `localhost`，所以很多安全限制就不会触发。

这也是为什么 SSH 隧道是最省心的办法。它不仅安全，而且能绕开浏览器对“HTTP + 公网 IP”的限制。

## 3. 公网访问问题

很多人第一次都会这样试：

```text
http://服务器公网IP:18789
```

然后控制台报错：

```text
control ui requires device identity (use HTTPS or localhost secure context)
```

这个报错非常关键，必须真正理解。

### 浏览器安全策略限制

问题的核心不是端口没开，不是 token 错，也不是 OpenClaw 只允许本机访问，而是浏览器安全上下文限制。

浏览器对这几类场景的态度大致是：

- `localhost`：安全上下文，允许
- `HTTPS`：安全上下文，允许
- `HTTP + 公网 IP`：非安全上下文，不允许

也就是说，哪怕你的 token 完全正确，只要你是通过 `http://公网IP:18789` 这样的方式直接打开，浏览器仍然可能拒绝某些需要设备身份或更高安全级别的能力。

### token 无法解决该问题

这是另一个特别容易误判的坑。

很多人看到登录失败，第一反应是：

- token 填错了
- token 过期了
- OpenClaw 没读到配置

但这个问题里，token 不是决定因素。因为浏览器在更前面那一层就已经把上下文判定为不安全了。也就是说，token 还没来得及“发挥作用”，页面能力就先被限制住了。

所以结论很明确：

`token 正确` 不等于 `HTTP 公网访问可用`

## 4. 解决方案

### 方案 1：SSH 隧道

这是最简单的方案，适合个人使用和临时管理。

```bash
ssh -L 18789:127.0.0.1:18789 root@服务器IP
```

然后本地访问：

```text
http://localhost:18789
```

优点是配置简单、安全性高、几乎不需要额外组件。

### 方案 2：HTTPS + 域名

如果你希望长期、稳定、从浏览器直接访问控制台，更推荐：

- 配域名
- 用 Nginx 做反向代理
- 给域名配置 HTTPS 证书

核心链路如下：

```text
浏览器 -> https -> nginx -> http://127.0.0.1:18789
```

这里的重点是：浏览器面对的是 HTTPS 安全上下文，而不是裸的 `http://公网IP:18789`。

Nginx 在中间做一层反代，把外部的 HTTPS 请求转发给本机的 OpenClaw Gateway。这样既满足浏览器安全策略，也避免直接把内部控制台裸暴露在公网端口上。

# 六、阿里云配置

如果你的服务器在阿里云上，还需要确认网络层面本身没拦住。

## 安全组开放端口 18789

如果你确实要测试端口直连，或者要给 Nginx 之外的链路排障，首先检查阿里云安全组里是否开放了相关端口。

OpenClaw 默认端口是：

```text
18789
```

如果安全组没放行，即使 Gateway 在服务器本机已经监听，外部仍然访问不到。

## 检查端口

在服务器里执行：

```bash
ss -tlnp | grep 18789
```

如果有输出，说明本机上确实有进程在监听这个端口。

这里要注意：`ss` 能看到监听，不代表公网一定能访问；它只能证明“服务在本机活着”。公网能不能通，还取决于安全组、系统防火墙、绑定地址、Nginx 配置等外层条件。

# 七、常见问题总结

这一部分是整个过程里最有价值的地方。很多问题第一次看像玄学，第二次再遇到其实已经有固定排查路径了。

## 1. VSCode 关闭导致服务断开

现象：

- 刚部署时一切正常
- 关闭 VSCode 或 SSH 后，网页和微信都失效

根因：

你是用 `openclaw gateway` 前台跑的，进程跟终端生命周期绑定。

解决：

改成 `systemd --user` 常驻，并开启 linger。

## 2. 网页打不开，其实是 SSH 隧道断了

现象：

- 昨天 `http://localhost:18789` 能打开
- 今天突然不行

根因：

你访问的是本地 `localhost`，真正依赖的是 `ssh -L` 建立的隧道。隧道一断，浏览器当然打不开，但服务本身未必挂了。

解决：

重新建立 SSH 隧道，再访问本地 `localhost`。

## 3. token 正确但无法登录

现象：

- token 没填错
- 页面还是报无法完成控制台登录或设备身份相关错误

根因：

问题不在 token，而在浏览器安全上下文。`HTTP + 公网 IP` 不满足要求。

解决：

改用 SSH 隧道，或者上 HTTPS + 域名。

## 4. 插件 rate limit 429

现象：

- 插件安装、模型调用或某些接口请求时返回 `429`

根因：

通常是 provider 限流、接口额度不足，或者短时间内请求太密。

解决：

- 降低请求频率
- 换更稳的主模型
- 配置 fallback
- 检查对应平台配额

## 5. `channelConfigs` 报错，但不影响运行

现象：

- 日志里看到和 `channelConfigs` 相关的报错或警告
- 但实际上 Gateway 和微信通道还能继续工作

理解方式：

这类报错不一定是致命错误。有些是配置字段兼容性、插件读取时机、旧配置残留导致的日志噪音。

建议：

先看功能是否真的不可用，不要一看到红字就立刻判断整套系统坏了。优先以以下结果为准：

- Gateway 是否仍在监听
- `openclaw channels list` 是否正常
- 微信是否能实际收发消息

## 6. `gateway already running`

现象：

再次执行 `openclaw gateway` 时提示：

```text
gateway already running
```

根因：

通常说明后台已经有一个 Gateway 实例在跑了，常见于你已经用 systemd 启动过，又手动再起一次。

解决：

先确认当前实例状态，而不是盲目重复启动。可以从 systemd 状态、端口监听、控制台可访问性几个方向检查。

# 八、安全提醒

OpenClaw 控制台不适合裸暴露在公网。

原因很现实：

- 控制台本身具备较高权限
- token 一旦泄露，风险很高
- 如果再叠加弱口令、HTTP 明文、开放来源过宽，问题会被放大

几个建议很重要：

- 使用 HTTPS
- 限制来源 IP
- 尽量通过 SSH 隧道或内网方式访问
- 不要把控制台长期公开暴露在公网

如果一定要公网可访问，至少应该通过 Nginx、HTTPS、访问控制、服务器安全组等多层限制一起做，不要直接开放一个裸的 `18789` 给全世界扫。

# 九、最佳实践总结

如果目标是“稳定可用，而不是刚好跑起来一次”，我更推荐下面这套组合。

## 推荐部署架构

- Gateway：用 `systemd` 常驻
- 访问：优先 HTTPS，临时管理可用 SSH 隧道
- 微信：通过 `channel` 接入
- 模型：主模型用 `flash` 类快速模型，配一个更稳的 fallback

可以把它理解成这样一条链路：

```text
OpenClaw Gateway(systemd 常驻)
    -> 模型主配置(flash)
    -> fallback 兜底
    -> openclaw-weixin channel
    -> HTTPS 或 SSH 隧道访问控制台
```

这套结构的好处是：

- 服务不会因为 VSCode 断开就停止
- 浏览器访问不会卡在安全上下文问题上
- 微信接入更稳定
- 主模型快，备用模型兜底，体验和稳定性更平衡

总结来说，OpenClaw 真正难的地方不是“安装成功”，而是把它放进一个长期可运行的环境里。只要把 Gateway 常驻、浏览器安全上下文、微信 channel、模型 fallback 这几件事理顺，后面的使用会顺很多。
